# HAL Recovery Compiler

![Python 3.11+](https://img.shields.io/badge/Python-3.11%2B-3776AB?logo=python&logoColor=white)

**Release:** `v0.3.1-public-poc`  
**Package version:** `0.3.1`

HAL Recovery Compiler is a simulation-only functional recovery profile compiler for defect-aware, route-aware workload placement on imperfect chip maps.

## Status

- **Status:** `v0.3.1-public-poc`
- **Scope:** simulation-only compiler
- **Runtime:** out of scope
- **Hardware control:** disabled
- **Certification:** none

## Core philosophy

**Physical yield may fail, but functional yield can still be recovered.**

**HAL connects what still works.**

## What it does

HAL Recovery Compiler:

- reads a synthetic chip defect map;
- reads workload requirements and explicit safety constraints;
- finds connected, four-neighbor, routable tile groups;
- generates candidate recovery profiles and functional passports;
- verifies generated artifacts against their source inputs;
- compares strict, placement-first, and route-aware recovery modes;
- recommends only a mode that passes the safe-mode requirements.

The default compiler remains route-aware. It may leave a workload unassigned when no connected and safely routable placement exists.

## What it does not do

This project:

- does not repair silicon;
- does not certify chips;
- does not control memory controllers;
- does not flash firmware;
- does not modify voltage or timing;
- does not operate hardware;
- does not replace human review;
- does not include Recovery Runtime.

It also contains no cloud service, external API integration, hardware driver, AI model, MILP, CP-SAT, or SQBM solver.

## Three-layer HAL structure

```text
YieldOS Shadow
= What still works?

HAL Recovery Compiler
= Where should roles and workloads be safely reassigned?

HAL Recovery Runtime
= Future separate project that may consume recovery_profile.json.
```

Recovery Runtime is intentionally out of scope for this repository. The compiler never loads a profile into hardware or firmware.

## Quick start

Requires Python 3.11 or later. Run these commands from the project root:

```bash
python -m pip install -e .
```

`validate` checks input structure and the hard safety boundary without solving or writing artifacts.

```bash
hal-rc validate samples/chip_001.json --workloads samples/workloads.json --constraints samples/constraints.json

hal-rc compile samples/chip_001.json --workloads samples/workloads.json --constraints samples/constraints.json --out artifacts/chip_001 --self-check --comparison-report

hal-rc verify-artifacts artifacts/chip_001 --chip samples/chip_001.json --workloads samples/workloads.json --constraints samples/constraints.json
```

The compile command produces a candidate profile, validates its invariants, compares recovery policies, and emits human-readable reports. It does not execute recovery.

For development tests:

```bash
python -m pip install -e ".[test]"
python -m pytest -q
```

## Expected demo result

- `route_aware` should be recommended when it has zero safety violations and complete accepted routes.
- `no_route_awareness` may assign more workloads but can be unsafe because of incomplete routes.
- Unassigned workloads are a safety feature, not automatically a failure.

## Optional commands

Compare the three deterministic policies:

```bash
hal-rc compare-baselines samples/chip_001.json --workloads samples/workloads.json --constraints samples/constraints.json --out artifacts/chip_001_comparison
```

Compare two compile runs while ignoring measured runtime telemetry:

```bash
hal-rc compare-artifacts artifacts/run_a artifacts/run_b
```

All commands are also available as `python -m hal_rc.cli ...`.

## Outputs

- `recovery_profile.json` — candidate workload assignments, disabled tiles, and preferred routes.
- `functional_passport.json` — functional-yield summary, remaining and blocked roles, and evidence.
- `solver_report.json` — route-aware placement, candidate-group, routing, and bounded local-search telemetry.
- `summary_report.html` — human-readable chip, assignment, verification, and recommendation summary.
- `artifact_validation_report.json` — invariant pass/fail report generated by `--self-check`.
- `comparison_report.json` — baseline metrics, trade-offs, and safe recommendation fields.
- `comparison_report.html` — human-readable baseline and recommendation report.

Generated `artifacts/` are excluded from source releases. Recreate them with the commands above.

See [docs/OUTPUTS.md](docs/OUTPUTS.md) for field-level intent.

## Recommended mode: raw score is not enough

The highest raw functional-yield score is descriptive, not prescriptive. A recommended mode must:

- have zero safety violations;
- have zero incomplete accepted routes;
- pass its invariants;
- preserve `hardware_control_enabled: false`;
- preserve `claim_boundary: simulation_only_not_certified`.

`no_route_awareness` can assign more workloads and score highly while accepting fragmented or incomplete routes. `route_aware` may assign fewer workloads, but is preferred when it preserves complete routes and the safety boundary.

An unassigned workload is evidence that the compiler refused an invalid candidate placement. It is not necessarily a compiler failure.

## Safety boundary

HAL Recovery Compiler does not repair silicon, certify chips, control memory controllers, flash firmware, or modify hardware. It generates candidate simulation-only recovery profiles for human review.

Every generated JSON report preserves:

```json
{
  "hardware_control_enabled": false,
  "human_review_required": true,
  "claim_boundary": "simulation_only_not_certified"
}
```

Read [docs/SAFETY_BOUNDARY.md](docs/SAFETY_BOUNDARY.md) and [docs/CLAIMS.md](docs/CLAIMS.md) before evaluating outputs.
